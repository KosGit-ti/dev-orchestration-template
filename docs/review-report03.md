## 3. ⚠️ 問題のある箇所

### 3-1. §3-1 のコードスニペット誤表記と行番号断定の問題

#### コードスニペットの誤表記

`review-report02.md §3-1` に掲載されたコードスニペット：

```
L65  SECRET_PATTERNS = [
         r"sk-[A-Za-z0-9]{32,}",  # OpenAI API key
         ...
     ]
```

これは **誤り**。実際のコード（`ci/policy_check.py`）：

```
61→SECRET_PATTERNS: list[str] = [
62→    r"AKIA[0-9A-Z]{16}",             # AWS Access Key ID
63→    r"-----BEGIN\s+...PRIVATE KEY-----",  # SSH 秘密鍵
64→    r"ghp_[A-Za-z0-9_]{36,}",        # GitHub Personal Access Token
65→    r"sk-[A-Za-z0-9]{32,}",          # 汎用 API キー
66→]
```

- L61 が `SECRET_PATTERNS: list[str] = [`（リスト定義開始）
- L65 が `r"sk-[A-Za-z0-9]{32,}",`（sk- パターン本体）
- L66 が `]`（リスト終端）

スニペットが L65 を「リスト定義の開始行」として誤表示していた。

#### 行番号断定の評価

ユーザーの懸念「:65 は SECRET_PATTERNS の開始行であり、sk- パターン本体は L66 以降では」は、
上記スニペット誤表記から生じた誤解である。実際には L65 が sk- パターン本体の行であり、
**`review-report01.md` の `:65` は正確**。

「:69 は個別パターンを指す可能性」については、L66 がリスト終端 `]` であり L69 は
`URL_PATTERN` の定義開始行であるため、この説は成立しない。

ユーザーが「一方を誤りと断定した点は論理不十分」と指摘した根拠となったスニペット自体が
誤表記であったため、断定の論理は（結果的に）正しかった。ただしスニペット誤表記は
`review-report02.md` の記述上の瑕疵として残る。

### 3-2. §3-1 の SCAN_DIRS 行番号評価の不正確さ

`review-report01.md` の記述: `ci/policy_check.py:22-26`
`review-report02.md` の評価: 「L22 はコメント行、定義本体は L23-27、コメント行を含む微差」

実際のコード（`ci/policy_check.py` L18-29）：

```
22→REPO_ROOT = Path(__file__).resolve().parent.parent
23→
24→# スキャン対象ディレクトリ（プロジェクトに合わせて変更）
25→SCAN_DIRS = [
26→    REPO_ROOT / "src",
27→    REPO_ROOT / "tests",
28→    REPO_ROOT / "scripts",
29→]
```

`review-report02.md` の評価に含まれる誤り：

| 問題点 | review-report02.md の記述 | 実際 |
|---|---|---|
| L22 の内容 | 「コメント行」 | `REPO_ROOT = Path(...)` （SCAN_DIRS と無関係な代入文） |
| 定義本体の開始行 | L23 | L25（3行ずれ） |
| 定義本体の終端行 | L27 | L29（2行ずれ） |
| review-report01.md のずれ幅 | 「終端も1つずれ」と示唆 | 開始3行・終端3行のずれ |

ユーザーの「終端行番号のずれへの言及不足」という指摘は正確であり、
さらに L22 をコメント行と誤認した点も問題である。
`review-report01.md` の `:22-26` という記述は、開始・終端ともに3行ずれており
「コメント行を含む微差」では説明できない誤りである。

| 存在しない | python 以外 | 生成されない ❌ |
